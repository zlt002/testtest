# Markdown 预览视图图片插入增强设计

## 背景

当前 Markdown 预览视图仅允许在少量元素上插入图片：

- 段落 `P`
- 标题 `H1-H6`
- 列表项 `LI`
- 表格单元格 `TD/TH`
- 引用块 `BLOCKQUOTE`

这套规则实现简单，但体验偏“技术定位”，不符合用户对文档编辑的直觉。用户在 PRD、方案文档等场景下，更希望像 Word 一样，在正文的大多数区域都能插图，而不是必须命中特定标签。

## 目标

将 Markdown 预览视图的图片插入能力升级为“正文大部分区域可插入”，在保证源码写入安全性的前提下，显著提升可用性。

本次目标：

- 在 Markdown 预览模式下，正文大部分位置都允许插图
- 插图行为优先符合用户视觉理解，而不是 DOM 标签限制
- 对无法精确映射源码的位置提供稳定降级策略
- 保留少量明确禁区，避免错误写入

本次不做：

- 不实现真正的富文本光标级编辑
- 不支持把图片插入代码块、Mermaid 图内部、行内代码中
- 不做完整 Markdown AST 到屏幕坐标的双向精确映射系统

## 用户体验设计

### 交互入口

保留现有两种入口：

- 顶部工具栏“图片”按钮
- `Ctrl + 鼠标悬停` 时出现的悬浮插图按钮

两种入口统一走同一套“插图目标解析”逻辑，避免行为不一致。

### 允许插图的总体规则

新的规则从“只允许部分标签”调整为“只要点击发生在正文 article 内，就尽量允许插图”。

默认允许的区域包括：

- 普通段落
- 标题区域
- 列表整体及列表项
- 引用块
- 表格区域
- 图片块附近
- 空白但仍属于某个正文块的区域
- 常见块级容器中的文本区域

### 保留禁区

以下区域仍然不允许插图：

- 代码块 `pre/code`
- Mermaid 渲染块
- 预览层浮动按钮、标注浮层等非正文交互 UI
- 无法归属到正文 article 的点击

### 插入位置规则

用户点击或悬停正文区域后，系统按以下优先级推断插入位置。

1. 最近可识别正文块后插入

优先找到最近的“块级锚点元素”，将图片 Markdown 插入到该块后方。块级锚点包括：

- `p`
- `h1-h6`
- `li`
- `blockquote`
- `table`
- `thead`
- `tbody`
- `tr`
- `ul`
- `ol`
- `hr`

2. 表格统一按“表格后插入”

如果点击发生在表格内部，无论最终命中的是 `table`、`tbody`、`tr`、`td`、`th`，都统一将图片插入到整张表格对应 Markdown 片段之后，而不是插到某个单元格行尾。

这样更符合文档编辑直觉，也能避免破坏表格结构。

3. 列表容器按“最近列表项后插入”

如果命中列表容器但无法稳定定位到具体 `li`，则退化为插入到该列表块末尾。

4. 空白区域按“最近上一个正文块后插入”

如果点击发生在视觉空白区域，但该位置仍位于 article 内，则使用最近的上一个正文块作为插入锚点。

这使得章节之间、段落留白附近的点击也能工作，体验更接近 Word。

5. 无法精确命中时按“当前章节末尾”降级

如果上面都失败，则尝试寻找最近的标题块，将图片插入该标题所属章节的最后一个已识别正文块之后。

6. 最终兜底：文档末尾插入

如果仍无法安全映射，则插入到 Markdown 文档末尾。

只有在检测到明确禁区时，才返回“不支持插图”。

## 技术设计

### 现状问题

当前实现依赖：

- 从点击 DOM 向上查找少量允许标签
- 取该元素 `textContent`
- 在 Markdown 源码中查找唯一文本匹配
- 根据匹配结果计算插入 offset

这会导致几个问题：

- 只有少数标签被允许
- 表格只能落到单元格文本行尾
- 空白区域无法工作
- 重复文本容易失败
- 用户点击的“视觉位置”与源码插入位置不够一致

### 新的解析策略

将插图定位从“单标签命中”升级为“多级锚点解析 + 降级链路”。

核心思路：

1. 先判断是否命中禁区
2. 在 article 内寻找最近块级锚点
3. 尝试根据锚点文本映射源码区段
4. 若失败，按更宽松的块级/章节级规则降级
5. 最终可安全时落到文档末尾

### 建议新增的数据结构

新增或演进目标解析结果类型：

```ts
type MarkdownInsertResolution =
  | { ok: true; offset: number; strategy: string }
  | { ok: false; message: string };
```

其中 `strategy` 用于调试和测试，例如：

- `block-after`
- `table-after`
- `list-after`
- `previous-block-after`
- `section-end`
- `document-end`

### DOM 侧锚点查找

建议新增一个更宽松的块级锚点查找函数，职责包括：

- 判断禁区
- 返回最近命中的块级元素
- 支持从文本节点、内联元素、空白点击、嵌套容器中回溯
- 对表格、列表做归一化

建议优先识别的锚点类别：

- 标题块
- 段落块
- 列表项
- 列表容器
- 引用块
- 表格块
- 分隔线

### Markdown 源码定位策略

源码定位仍以“文本匹配”作为第一实现手段，但策略调整如下：

1. 块级精确匹配

对锚点文本做唯一匹配，成功后插入到对应块后面。

2. 结构化特判

对表格、列表、标题做独立处理：

- 表格：尽量匹配整段表格文本，失败则回退到表格结束行
- 列表：优先匹配具体项，失败则匹配整个列表片段末尾
- 标题：匹配标题行后，可继续查找下一个同级或更高级标题，推断章节范围

3. 邻近块降级

当前块无法唯一定位时，尝试使用前一个可识别正文块。

4. 章节降级

若当前点击处属于某个标题章节，则插入到该章节最后一个已识别块之后。

5. 文档末尾兜底

若没有可靠映射但不属于禁区，则插入文档末尾。

### 对现有接口的影响

重点修改文件预计为：

- `apps/extension/entrypoints/sidepanel/routes/file-preview.markdown-insert-position.ts`
- `apps/extension/entrypoints/sidepanel/routes/file-preview.tsx`
- 对应测试文件

`file-preview.tsx` 层面尽量不改 UI 结构，主要保持：

- 点击更新当前插图目标
- `Ctrl + 悬停` 更新悬浮插图按钮位置
- 工具栏插图按钮复用统一解析逻辑

## 风险与处理

### 风险一：重复文本导致插入位置不稳定

处理方式：

- 保留唯一匹配优先
- 增加相邻块、章节级、文档尾部等稳定降级链路
- 测试覆盖重复标题、重复段落、重复列表项

### 风险二：表格和列表结构被破坏

处理方式：

- 表格一律插到表格整体后方
- 列表尽量插在项后或列表块后，不拼接到行内

### 风险三：用户感知与实际落点不一致

处理方式：

- 优先遵循“视觉上最近的正文块”
- 保持悬浮按钮贴近将要插入的块显示
- 后续如有需要可补充轻量提示文案

### 风险四：兜底过宽导致“哪里都能插，但位置不准”

处理方式：

- 仅在 article 正文内启用宽松策略
- 明确保留禁区
- 优先最近块、其次章节、最后文档末尾，保证行为可预测

## 测试设计

需要补充和调整测试覆盖：

### 单元测试

- 段落、标题、引用、列表、表格仍可插
- 点击表格内部时，统一落到表格后
- 点击列表容器空白区域时，可落到列表末尾
- 点击章节空白区域时，可落到最近正文块后
- 当前块重复文本时，能降级到相邻块或章节末尾
- 禁区内仍返回不支持
- 无法定位但在正文内时，落到文档末尾

### 交互测试

- 工具栏“图片”按钮在宽松规则下能正常触发
- `Ctrl + 悬停` 在更多正文区域出现插图按钮
- 粘贴图片仍复用统一目标解析逻辑

## 分阶段实施建议

第一阶段：

- 放宽允许区域
- 表格改为“整表后插”
- 增加空白区域和相邻块降级
- 增加文档末尾兜底

第二阶段：

- 章节级定位优化
- 更稳的列表/表格片段匹配

## 结论

本方案选择“接近 Word 的体验 + 可控的源码兜底”路线：

- 在正文大部分区域都允许插图
- 继续屏蔽少数明确禁区
- 通过多级降级策略降低失败率
- 先不引入复杂 AST 映射系统

这能以较低复杂度显著改善 PRD / Markdown 文档预览编辑体验，并保持当前代码结构可演进。
