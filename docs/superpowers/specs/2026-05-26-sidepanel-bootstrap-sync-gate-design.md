# 侧边栏启动同步门禁设计

**背景**

当前扩展已经具备“点击扩展图标时触发一次远端 ACCR 同步”的能力，但如果把同步严格阻塞在图标点击瞬间，会直接拖慢 sidepanel 打开体验。对于真实用户来说，点击扩展图标后没有视觉反馈、长时间等待，会明显降低可用性。

现在的目标不是再把同步塞回 background 点击链路，而是调整为：

- 点击扩展图标后，sidepanel 先立即打开
- sidepanel 中央显示一个启动门禁页
- 门禁页负责并行执行：
  - 远端 skill / 配置同步
  - 模型配置检查
- 两个任务都完成后，再决定是否放开真正主界面

这样既保留“点击时触发同步”的时机，又避免把点击动作本身阻塞得太久。

**目标**

- 点击扩展图标后，sidepanel 秒开，不在 background 中长时间等待同步完成
- sidepanel 启动后显示全屏门禁态：“配置内容同步中...”
- 后台并行执行远端 ACCR 同步与模型配置检查
- 只有在“两者都完成”后，才允许进入正常主界面
- 如果模型配置检查失败，保留拦截态并引导用户去设置
- 如果同步失败，保留拦截态并允许重试

**非目标**

- 不改变远端 ACCR 同步核心逻辑
- 不在本次设计中重写模型设置页面
- 不把整个 sidepanel 重构成新的路由体系

## 方案对比

### 方案 A：继续在 background 点击时同步，完成后再打开 sidepanel

**优点**

- 状态最一致，打开时一定已经同步完成

**缺点**

- 点击图标时感知最差
- 网络、下载、备份稍慢就会让用户觉得扩展卡死

### 方案 B：sidepanel 立即打开，门禁页拦截直到启动检查完成

**优点**

- 用户有立即反馈
- 可以明确展示“同步中 / 检查中 / 失败原因”
- 与现有 sidepanel 架构更兼容

**缺点**

- 需要增加一层 sidepanel 启动态管理

### 方案 C：主界面直接打开，只禁用部分交互

**优点**

- 表面上加载更快

**缺点**

- 容易漏掉仍可点击的入口
- 页面状态会变得复杂，难以保证一致性

## 推荐方案

采用 **方案 B：sidepanel 立即打开，门禁页拦截**。

原因：

- 用户点击后能立刻看到反馈
- 可以把“远端同步”和“模型配置检查”清晰地展示出来
- 不需要把同步执行时间压缩到点击瞬间
- 后续失败处理、重试、跳转模型设置都更自然

## 交互设计

### 1. 启动门禁页

sidepanel 启动后，如果本次启动检查尚未完成，则不渲染正常主界面，而是显示一个全屏门禁态。

文案方向：

- 主标题：`配置内容同步中...`
- 副标题：`正在同步远端技能与检查模型配置`

状态展示建议：

- `技能同步中`
- `模型配置检查中`
- `技能同步完成`
- `模型配置检查完成`
- `技能同步失败`
- `模型配置检查失败`

### 2. 解锁条件

按你确认的规则，sidepanel 解锁条件是：

- `远端 skill 同步完成`
- `模型配置检查完成`

注意：

- 这里是“两者都完成”，不要求“两者都成功”
- 但如果模型配置检查结果显示当前不可用，则主界面仍然保持拦截态，并给出明确行动入口

### 3. 失败态

#### 同步失败

显示：

- 同步失败原因
- `重试同步` 按钮

行为：

- 点击后重新执行远端同步
- 模型配置检查结果可以复用最近一次结果，或按实现简化为一起重跑

#### 模型配置检查失败

显示：

- 检查结果说明
- `去模型设置` 按钮
- 可选保留 `重新检查` 按钮

行为：

- `去模型设置` 直接导航到 sidepanel 设置页中的模型设置区域

## 技术设计

### A. background 层

文件：

- `apps/extension/entrypoints/background/index.ts`

职责调整：

- 点击扩展图标时不再等待同步完成
- 只负责：
  - 打开 sidepanel
  - 可选记录“本次是 action click 触发”的启动上下文

原则：

- 同步不再是点击链路上的硬阻塞项

### B. sidepanel 启动门禁层

建议新增：

- `apps/extension/entrypoints/sidepanel/lib/bootstrap-gate.ts`
- `apps/extension/entrypoints/sidepanel/lib/bootstrap-gate.test.ts`

职责：

- 统一管理 sidepanel 启动时的门禁状态机
- 并行触发：
  - 远端同步请求
  - 模型配置检查请求
- 产出统一状态：
  - `loading`
  - `ready`
  - `sync_failed`
  - `model_check_failed`
  - `blocked`

### C. sidepanel UI 挂载点

优先挂在：

- `apps/extension/entrypoints/sidepanel/routes/__root.tsx`

原因：

- 这是整个 sidepanel 的根容器
- 在这里拦截最稳，不会出现某些子页面提前可见的问题

行为：

- 如果 bootstrap gate 未完成，渲染全屏门禁页
- 完成后才渲染原始路由内容

### D. 模型配置检查入口

候选位置：

- `apps/extension/entrypoints/sidepanel/lib/modelConfig.ts`
- 或 `apps/extension/entrypoints/sidepanel/lib/agent-v2/client.ts`

职责：

- 对接 `agent-backend-v2` 已有模型配置相关接口
- 获取模型配置运行时可用性
- 输出给 bootstrap gate 统一判断

### E. 后端同步接口

继续复用现有：

- `POST /api/accr-sync/run`

请求：

```json
{
  "mode": "remote",
  "force": true,
  "trigger": "extension-action-click"
}
```

说明：

- 启动门禁里的同步应继续使用 `force: true`
- 原因是需要修复用户手工改坏本地 `~/.claude` 的情况，不能只靠版本号跳过

## 状态机设计

建议的状态机：

1. `idle`
2. `running`
3. `sync_failed`
4. `model_check_failed`
5. `ready`

具体规则：

- 初始化进入 `running`
- 并行执行同步和模型检查
- 若同步失败，进入 `sync_failed`
- 若同步成功但模型配置检查显示不可用，进入 `model_check_failed`
- 若两者都完成且允许进入主界面，进入 `ready`

## 测试策略

### 单元测试

- bootstrap gate 并行执行两项任务
- 同步失败时进入失败态
- 模型配置检查失败时进入失败态
- 两项任务都完成时进入 ready
- 重试同步会重新触发同步流程

### 组件测试

- 根路由在 gate 未完成时显示全屏拦截页
- gate 完成后渲染原始 sidepanel 内容
- gate 失败时显示对应按钮和提示

### 集成测试

- 点击扩展图标后，sidepanel 能立即打开
- sidepanel 初始显示“配置内容同步中...”
- 同步成功且模型配置检查完成后进入主界面

## 风险与注意事项

### 1. 不要把 gate 状态塞进太多现有页面组件

门禁逻辑应集中在 root 层，否则会让多个路由各自判断“是否可用”，维护成本很高。

### 2. 不要因为模型配置检查失败而静默放行

既然主界面依赖模型能力，检查失败时应明确提示，而不是让用户进入一个表面可用、实际不可用的主界面。

### 3. 重试动作要避免多重并发

无论是同步还是模型检查，都应在 gate 内做单飞控制，避免连续点击造成重复请求。

## 成功标准

- 点击扩展图标后，sidepanel 立即打开
- sidepanel 中央显示启动门禁页
- 远端 skill 同步和模型配置检查并行执行
- 两者完成后才进入正常主界面
- 同步失败可重试
- 模型配置检查失败可跳转设置
