# 会话 Markdown 导出设计

## 背景

当前侧边栏聊天已经支持：

- 展示当前会话时间线
- 展示按运行聚合后的助手回答卡片
- 查看历史会话

但还缺少两个直接影响可用性的能力：

- 将当前整个会话导出为 Markdown 文档
- 对单个助手回答执行复制或导出 Markdown

用户已经明确本次范围：

- 先做纯对话版，不导出工具调用、思考过程、交互状态等过程信息
- Markdown 中保留每条消息时间

## 目标

为侧边栏聊天增加三项能力：

- 当前会话一键导出 `.md`
- 单条助手回答复制为 Markdown
- 单条助手回答导出为 `.md`

导出的内容应适合直接阅读、归档和分享，而不是内部调试日志。

## 非目标

本次不做以下内容：

- 不导出工具调用、thinking、interaction、run_status、error 细节
- 不做“完整记录版”和导出模式切换
- 不新增后端接口
- 不改造现有会话数据协议
- 不支持用户单条消息单独导出

## 设计结论

### 1. 新增统一的聊天 Markdown 导出层

新增一个专用导出模块，例如：

- `apps/extension/entrypoints/sidepanel/lib/chat-markdown-export.ts`

这个模块统一负责：

- 从当前聊天 UI 可用的数据结构生成 Markdown
- 格式化角色标题
- 格式化时间
- 生成默认文件名

这样“整会话导出”和“单条回答复制/导出”都复用同一套规则，避免后续多个地方各写一份字符串拼接逻辑。

### 2. 导出数据来源

整会话导出直接复用当前聊天页已经加载的时间线数据，不新增单独的导出查询。

具体上：

- 用户消息来自 `ConversationRunItem` 中的 `user.message`
- 助手消息来自 `RunCard.responseMessages`
- 当某个 `RunCard` 没有 `responseMessages` 时，使用当前界面已经采用的 fallback 文本规则

导出层只提取“最终可读文本”，不带过程预览、文件引用、todo、工具状态。

### 3. Markdown 结构

整会话导出建议格式如下：

```md
# 会话记录

> 会话标题：xxx
> 会话 ID：xxx
> 导出时间：2026-05-28 12:30:00

## 用户 · 2026-05-28 12:00:00

用户消息正文

## 助手 · 2026-05-28 12:00:08

助手回答正文
```

规则：

- 顶部包含会话标题、会话 ID、导出时间
- 每条消息用二级标题表示角色和消息时间
- 消息正文之间保留空行，确保 Markdown 可读性
- 时间统一使用本地格式化后的可读字符串

### 4. 单条助手回答结构

单条助手回答复制/导出时使用更轻量的格式：

```md
## 助手 · 2026-05-28 12:00:08

助手回答正文
```

如果该回答是由多个 `responseMessages` 片段组成，则按当前显示顺序拼接成一条完整回答，中间保留空行。

### 5. 入口位置

整会话导出入口放在聊天页当前会话操作区，属于会话级操作。

建议形式：

- 会话顶部更多操作菜单增加“导出 Markdown”

单条助手回答入口放在助手回答卡片头部或卡片右上角操作区。

建议提供两个动作：

- `复制 Markdown`
- `导出 Markdown`

这样不会影响当前消息正文阅读区，也方便与后续更多消息操作并列扩展。

### 6. 文件名规则

整会话导出默认文件名建议包含：

- 会话标题
- 会话时间或导出时间

示例：

- `会话记录-客户管理列表-2026-05-28-120000.md`

单条助手回答导出默认文件名建议包含：

- `助手回答`
- 回答时间

示例：

- `助手回答-2026-05-28-120008.md`

文件名中需要做非法字符清洗，避免浏览器下载失败。

### 7. 复制与下载能力

复制能力优先使用浏览器剪贴板：

- `navigator.clipboard.writeText(markdown)`

导出能力复用现有前端下载方式，直接生成 Markdown Blob 并触发下载。

如果项目中已有统一下载辅助函数，则优先复用现有实现。

## 交互规则

### 会话导出

- 用户点击“导出 Markdown”后立即开始下载
- 下载成功后给出轻量成功提示
- 失败时给出错误提示

### 单条回答复制

- 用户点击“复制 Markdown”后，将该条回答的 Markdown 写入剪贴板
- 成功时给出“已复制”反馈
- 失败时给出错误提示

### 单条回答导出

- 用户点击“导出 Markdown”后，下载对应 `.md` 文件
- 成功时给出轻量反馈
- 失败时给出错误提示

## 实现建议

### 数据转换边界

建议先把聊天 UI 的时间线项转换为一个更适合导出的中间结构，例如：

- `role`
- `timestamp`
- `text`

这样 UI 组件只负责触发动作，Markdown 拼接逻辑集中在导出模块。

### 组件改动边界

建议改动以下位置：

- `routes/chat.index.tsx`
  - 接入整会话导出入口
  - 给助手回答卡片注入单条导出操作
- `components/chat/AssistantBubble.tsx`
  - 如有必要，承载轻量操作插槽；若卡片层更合适，则保持不动
- 新增导出模块
  - 统一封装 Markdown 生成和文件名处理

尽量不要改动 `UserBubble` 的现有折叠和上下文显示逻辑。

## 测试要求

需要补充测试覆盖以下场景：

- 整会话导出只包含用户与助手正文，不包含 tool/thinking/run_status
- 导出结果包含每条消息时间
- 单条助手回答复制生成的 Markdown 格式正确
- 单条助手回答导出文件名符合规则
- 多段 `responseMessages` 会按顺序合并
- `RunCard` 无 `responseMessages` 时采用 fallback 文本

## 风险与取舍

### 1. 当前 UI 是按运行卡片聚合，不是原始消息逐条平铺

这意味着“单条助手回答”更接近“单个运行卡片的最终回答”，而不是原始事件流中的一条 message。

本次选择跟随当前 UI 认知模型：

- 用户看到哪张助手回答卡片，就对哪张卡片执行复制/导出

### 2. 纯对话版会丢失过程信息

这是本次明确接受的取舍，因为用户当前目标是可读、可分享的 Markdown，而不是调试审计日志。

后续如果需要完整记录版，可以在当前统一导出层上继续扩展。
