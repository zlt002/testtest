# Current Page Codebase

适用场景：用户正在问“当前页面这块代码在哪、怎么改、这个按钮对应哪个接口、这个页面问题更像前端还是后端”这类和当前页面联动分析直接相关的问题。

固定流程：

1. 先调用 `resolve_current_page_codebase_context`。
2. 把返回结果当成“页面证据采集结果”来消费，不要再假设它已经完成业务归属、知识库选择或 graph 项目分组。
3. 第一层先向用户说明当前拿到了哪些证据：
   - URL / pathname / hashRoute
   - 页面标题和页面文本摘要
   - API candidates
   - resource hints / frame hints
4. 第二层再根据用户问题决定是否需要继续补证据：
   - 证据足够：直接进入代码定位或改动分析
   - 证据不足：明确告诉用户缺什么，例如接口名、DOM 区域、交互路径、网络请求
5. 第三层如果真的要落到代码定位，再动态判断更像哪一层：
   - 前端页面 / DOM / 交互 / 样式
   - 后端接口 / 服务编排 / 数据处理
   - 共享组件 / hooks / utils / 设计系统
6. 源码来源判断优先使用 skill 自己维护的规则文件或外部可更新规则；不要依赖扩展设置页里的旧“页面图谱路由”结论。
7. graph 查询优先使用 `codebase-memory-mcp` 的 `search_graph`、`trace_path`、`get_code_snippet`，并且只围绕当前页面证据和用户明确提到的对象做最短路径分析。
8. 如果证据只能支持“候选定位”，必须明确说明这是候选，不要伪造成确定归属。

输出顺序建议：

1. 页面证据
2. 候选接口 / 候选入口
3. 代码定位判断
4. 改动方案
5. 风险与验证

工作要求：

- 页面证据阶段只说事实，不要提前输出“命中了哪条规则”“建议知识库”“建议查询模式”。
- 如果用户继续追问“代码在哪 / 怎么改”，再决定是否查 codebase-memory、知识库或进一步交互补证据。
- 当需要做来源判断时，优先复用可维护规则；规则应该允许本地更新，而不是写死在页面分析主流程里。
- 如果当前页面证据和用户问题之间仍有明显断层，要先说明限制，再给出下一步最小补证据动作。
