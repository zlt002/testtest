# Current Page Codebase

适用场景：用户正在问“当前页面这个按钮查什么知识库更准、建议用什么 `/ewankb-server-query` 命令、这个页面对应哪个接口、后续代码该往哪一层追”这类和当前页面联动分析直接相关的问题。

当前页面分析的默认目标，已经不是直接把整段 markdown 塞进输入框，而是先产出：

1. 紧凑证据卡片
2. 单独的 `/ewankb-server-query ...` 建议命令
3. 兼容旧链路的 `chatSummary.markdown`

固定流程：

1. 先调用页面分析相关能力，优先消费结构化结果：
   - `analysisCard`
   - `suggestedCommand`
   - `evidence`
   - `chatSummary.markdown`
2. 第一层先围绕证据卡片说明页面事实：
   - 页面名称 / 路由
   - 目标操作
   - 推断意图
   - 表头或关键业务字段
   - 候选接口与置信度
3. 第二层优先给出建议命令，命令必须单独展示，且必须以 `/ewankb-server-query` 开头。
4. 如果用户只是想“查知识库”，优先围绕建议命令继续，而不是重复展开整页原始文本。
5. 如果用户继续追问“代码在哪 / 怎么改 / 谁调用这个接口”，再进入代码定位或代码穿透。
6. 做代码定位时，优先使用 `codebase-memory-mcp` 的 `search_graph`、`trace_path`、`get_code_snippet`，并且只围绕页面证据和用户明确提到的对象做最短路径分析。
7. 如果证据只能支持“候选判断”，必须明确说是候选，不要伪造成确定归属。

输出顺序建议：

1. 页面证据卡片
2. 建议命令
3. 候选接口 / 候选入口
4. 代码定位判断
5. 风险与验证

工作要求：

- 页面分析阶段优先产出“高价值、可检索”的短词，不要把整页长文本原样拼进查询。
- 建议命令的查询词优先级应保持为：页面功能词、操作意图词、接口技术词、业务字段词。
- `suggestedCommand` 和 `chatSummary.markdown` 是两个不同用途的输出：前者用于插入查询，后者只做兼容摘要。
- 如果用户继续追问“代码在哪 / 怎么改”，再决定是否查 codebase-memory、知识库或进一步补证据。
- 如果当前页面证据和用户问题之间仍有明显断层，要先说明限制，再给出下一步最小补证据动作。
